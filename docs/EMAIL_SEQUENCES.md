## 5. IMPLEMENTATION NOTES

### Triggers & Conditions

All engagement sequences are triggered server-side based on user behavior data:

- **Low Activity**: No login events in 72 hours
- **No Watchlist**: Watchlist count = 0 after Day 2
- **No Alerts**: Alert count = 0 after Day 3
- **Inactivity Re-engagement**: No login in 7+ days; escalates messaging

### Timing Considerations

- All times respect user timezone (parsed at signup)
- Stagger sends by 2-4 hours within cohorts (reduces delivery load)
- A/B test subject lines at 50/50 split by default
- Track opens, clicks, unsubscribes, bounces

### Copy Principles

1. **Short**: Most emails are under 100 words body copy
2. **Specific**: Feature-focused, not marketing fluff
3. **Action-oriented**: Every email has one clear CTA
4. **Trader voice**: Jargon OK, but no corporate speak
5. **Directness**: "We read replies" builds trust faster than promises

### Metrics to Track

- **Welcome sequence**: Watchlist adoption rate, alert setup rate, feature engagement by email
- **Engagement sequence**: Re-activation rate per trigger type
- **Feedback sequence**: NPS score distribution, feature request volume, bug report actionability
- **Upgrade sequence**: Conversion rate to Pro, early bird lock-in %

### Personalization Tokens

- `[First Name]`: User's first name
- `[Biggest mover from your watchlist]`: Ticker + % change (top 1 from user's watchlist)
- `[Secondary mover]`: Ticker + % change (2nd biggest)
- `[X weeks before beta close]`: Dynamic count-down
- `[DATE]`: Beta close date

### Unsubscribe & Preferences

- All emails include unsubscribe link (legal requirement)
- Provide preference center: onboarding vs. engagement vs. upgrade sequences can be toggled independently
- Users can change notification frequency (daily digest vs. real-time)
